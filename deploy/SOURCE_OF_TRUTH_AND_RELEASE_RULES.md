# قواعد المصدر والإصدار

هذا الملف هو المرجع الصارم لأي وكيل أو مطور يعمل على المشروع.

## 1. مصدر الحقيقة

- `GitHub` هو المصدر الرسمي الوحيد للكود وملفات `deploy`.
- `GHCR` هو مخزن صور التشغيل فقط.
- أي متجر داخل `/opt/client-stores/*` هو بيئة تشغيل عميل، وليس مصدرًا رسميًا.

## 2. أين يبدأ أي تعديل

- أي تعديل يبدأ من ملفات المشروع داخل المستودع `express-trade-kit`.
- لا يجوز البدء من `GHCR`.
- لا يجوز اعتبار ملفات العميل الحية في `/opt` أساسًا للتطوير.

## 3. إذا حصل إصلاح حي

- يمكن تطبيق إصلاح عاجل على البيئة الحية عند الضرورة.
- لكن لا يعتبر الإصلاح منجزًا حتى يعود إلى ملفات المشروع الأصلية.
- بعد ذلك يجب:
  1. اختبار التعديل.
  2. `commit`
  3. `push` إلى GitHub
  4. بناء release جديدة إذا كانت الصور تأثرت.

## 4. متى نبني صورة جديدة

- نبني صورة جديدة فقط إذا تغير:
  - كود `api`
  - كود `web`
  - أو شيء يدخل داخل صورة Docker نفسها
- إذا كان التغيير فقط في:
  - ملفات `deploy`
  - ملفات `Caddy`
  - سكربتات التشغيل
  - التوثيق
  فلا يلزم release جديدة في `GHCR` إلا إذا قررنا إصدارًا تشغيليًا جديدًا عمدًا.

## 5. التسلسل الإجباري الصحيح

```text
repo -> test -> commit -> push -> build image -> push GHCR -> deploy
```

لا يجوز عكس هذا الترتيب.

## 6. ما الذي لا يجوز اعتباره متزامنًا تلقائيًا

- `GitHub` و`GHCR` ليسا متزامنين تلقائيًا.
- `GitHub` وبيئات العملاء الحية ليسا متزامنين تلقائيًا.
- التزامن لا يعتبر صحيحًا إلا بعد التحقق الصريح.

## 7. التحقق قبل إعلان أي نسخة

قبل القول إن النسخة جاهزة:

1. تأكد أن التعديل موجود في المستودع المحلي.
2. تأكد أنه دُفع إلى GitHub.
3. إذا كانت الصور تأثرت، تأكد أن build الصحيح تم من داخل مجلد المشروع.
4. إذا كانت البيئة الحية تأثرت، تأكد أن العميل يستعمل الإصدار الجديد فعلًا.

## 8. القاعدة النهائية

- المشروع الأساسي هو الأصل.
- `Golden Image` ناتج من الأصل.
- متاجر العملاء نسخ تشغيلية من هذا الناتج.

أي وكيل يخلط بين هذه الثلاثة يخرج عن مسار العمل الصحيح.
